# ADLC Framework

**Agentic Development Life Cycle** — a pipeline that takes a plain-English request all the way to
a pushed branch, pausing only for the two approvals that matter. It runs on **Claude Code, Cline,
Gemini CLI**, and any **`AGENTS.md`**-aware agent (Cursor, Copilot, Codex, Windsurf, Aider, …),
from **one provider-neutral source of truth**.

```
"add a health endpoint"
      │
 1 Ticket ─▶ 2 Spec ─▶ [GATE 1: approve] ─▶ 3 Code ─▶ 4 Tests ─▶ 5 Verify ─▶ [GATE 2: approve] ─▶ 6 Ship (git push)
                                                          ▲                        │
                                                          └──── fail: retry ───────┘
```

## Why it isn't locked to one LLM

The pipeline's "intelligence" is just four provider-neutral things, all under [`core/`](core/):

1. **Stage instructions** — plain markdown, one per role ([`core/stages/`](core/stages)).
2. **Deterministic scripts** — everything mechanical (ticket keys, state, Jira, git, compare
   URLs) lives in [`core/scripts/adlc`](core/scripts/adlc) + [`jira_ticket.py`](core/scripts/jira_ticket.py)
   and runs **identically under any model** — the LLM only does the spec/code/tests reasoning.
3. **Templates** — [`core/templates/`](core/templates).
4. **A shared state convention** — `docs/adlc/<KEY>/state.md`.

What differs per host is only *packaging*, so [`build.py`](build.py) **generates a thin adapter
for each host** from `core/`. Edit `core/`, re-run `build.py`, and every host re-syncs — no drift.

```
core/            # SINGLE SOURCE OF TRUTH — edit here
├── orchestrator.md          stages/*.md   skills/*.md   templates/*.md
└── scripts/{adlc, adlc.ps1, jira_ticket.py, lib.sh}

adapters/        # GENERATED by build.py — do NOT hand-edit
├── claude-code/   .claude-plugin/ + agents/ + skills/ + commands/   (real sub-agents)
├── cline/         .clinerules/ + .clinerules/workflows/adlc.md       (single-agent workflow)
├── gemini/        .gemini/commands/adlc.toml + GEMINI.md             (TOML slash command)
└── universal/     AGENTS.md                                          (cross-tool fallback)
```

Only Claude Code has real sub-agents, so there each stage is an isolated agent. On single-agent
hosts the **same** runbook is followed inline, one role at a time — identical instructions,
graceful degradation.

## Install

```bash
git clone <this-repo-url> adlc-framework
cd adlc-framework
./install.sh <host> [target-project-dir]      # macOS/Linux/Git Bash
#   or on Windows PowerShell:
./install.ps1 <host> [target-project-dir]
#   host = claude | cline | gemini | universal | all
```

- **claude** → registers + installs the plugin (`claude plugin marketplace add adapters/claude-code`).
- **cline** → copies `.clinerules/` into your project.
- **gemini** → copies `.gemini/` + `GEMINI.md` into your project.
- **universal** → copies `AGENTS.md` into your project.

The installer always runs `build.py` first. For non-Claude hosts, add the scripts to your PATH so
`adlc` resolves (the installer prints the exact `ADLC_HOME`/`PATH` lines).

## Configure (optional)

Jira is optional. With no config the framework uses **local ticket files** (`docs/adlc/tickets/`)
and the rest of the pipeline is identical. For real Jira Cloud, copy `.env.example` → `.env` or
export `JIRA_BASE_URL`, `JIRA_EMAIL`, `JIRA_API_TOKEN`, `JIRA_PROJECT_KEY`.

## Use

| Host | How to start |
|------|--------------|
| Claude Code | `/adlc "let users reset their password by email"` |
| Cline | invoke the `adlc` workflow with your request |
| Gemini CLI | `/adlc "..."` |
| AGENTS.md tools | tell the agent to "run the ADLC pipeline for: ..." |

Then follow the two prompts (approve the spec, approve the push). Re-run with `resume <KEY>` to
continue from `state.md`.

## The deterministic engine (works with zero LLM)

```bash
adlc init "add a health endpoint"     # -> ADLC-001, seeds docs/adlc/ADLC-001/
adlc next-key                          # next local key
adlc jira mode|pick|create ...         # dual-mode ticketing
adlc get-state <KEY> <field> | set-state <KEY> <field> <value>
adlc approve <KEY> gate1|gate2         # record a human approval
adlc compare-url <branch>              # PR compare URL from origin
adlc ship <KEY> [--no-push]            # commit (+push) + print compare URL
```

## Extending

- Change behavior in `core/`, run `py build.py`, commit. Never edit `adapters/`.
- Add a host: write a `build_<host>()` in `build.py` that emits that host's config from `core/`.
- Later: wrap `core/scripts` in an MCP server for MCP-capable hosts (structure is ready).

## Out of scope (for now)
- Automated GitHub **PR creation** (shipper prints the compare URL; open the PR manually).
- Dedicated Cursor/Windsurf adapters (covered by `AGENTS.md`).
- MCP server layer and CI wiring.
