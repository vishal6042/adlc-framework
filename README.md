# ADLC Framework

**Agentic Development Life Cycle** ‚Äî a pipeline that takes a plain-English request all the way to
a pushed branch, pausing only for the two approvals that matter. It runs on **Claude Code, Cline,
Gemini CLI**, and any **`AGENTS.md`**-aware agent (Cursor, Copilot, Codex, Windsurf, Aider, ‚Ä¶),
from **one provider-neutral source of truth**.


## Why it isn't locked to one LLM

The pipeline's "intelligence" is just four provider-neutral things, all under [`core/`](core/):

1. **Stage instructions** ‚Äî plain markdown, one per role ([`core/stages/`](core/stages)).
2. **Deterministic scripts** ‚Äî everything mechanical (ticket keys, state, Jira, git, compare
   URLs) lives in [`core/scripts/adlc`](core/scripts/adlc) + [`jira_ticket.py`](core/scripts/jira_ticket.py)
   and runs **identically under any model** ‚Äî the LLM only does the spec/code/tests reasoning.
3. **Templates** ‚Äî [`core/templates/`](core/templates).
4. **A shared state convention** ‚Äî `docs/adlc/<KEY>/state.md`.

What differs per host is only *packaging*, so [`build.py`](build.py) **generates a thin adapter
for each host** from `core/`. Edit `core/`, re-run `build.py`, and every host re-syncs ‚Äî no drift.

```
core/            # SINGLE SOURCE OF TRUTH ‚Äî edit here
‚îú‚îÄ‚îÄ orchestrator.md          stages/*.md   skills/*.md   templates/*.md
‚îî‚îÄ‚îÄ scripts/{adlc, adlc.ps1, jira_ticket.py, lib.sh}

adapters/        # GENERATED by build.py ‚Äî do NOT hand-edit
‚îú‚îÄ‚îÄ claude-code/   .claude-plugin/ + agents/ + skills/ + commands/   (real sub-agents)
‚îú‚îÄ‚îÄ cline/         .clinerules/ + .clinerules/workflows/adlc.md       (single-agent workflow)
‚îú‚îÄ‚îÄ gemini/        .gemini/commands/adlc.toml + GEMINI.md             (TOML slash command)
‚îî‚îÄ‚îÄ universal/     AGENTS.md                                          (cross-tool fallback)
```

Only Claude Code has real sub-agents, so there each stage is an isolated agent. On single-agent
hosts the **same** runbook is followed inline, one role at a time ‚Äî identical instructions,
graceful degradation.

## Install ‚Äî one shared copy, no marketplace

The framework stays in **one common place** (this clone). The installer puts its `core/scripts`
on your PATH once and drops each host's small instruction files into that host's **user-global**
config dir ‚Äî so **every project** can use it. Nothing is published to a marketplace or plugin
registry, **not even for Claude Code**.

```bash
git clone <this-repo-url> adlc-framework      # keep this folder ‚Äî it IS the install
cd adlc-framework
./install.sh <host> [target-project-dir]      # macOS/Linux/Git Bash
#   or on Windows PowerShell:
./install.ps1 <host> [target-project-dir]
#   host = claude | cline | gemini | universal | all
```

What each host installs (globally, for all projects):

- **claude** ‚Üí copies `agents/ skills/ commands/` into `~/.claude/`. `/adlc` appears in every project. No plugin, no marketplace.
- **cline** ‚Üí installs the rule + workflow into the global `~/Documents/Cline/{Rules,Workflows}/`.
- **gemini** ‚Üí installs the user command `~/.gemini/commands/adlc.toml` (+ `~/.gemini/GEMINI.md`).
- **universal** ‚Üí copies `AGENTS.md` into the target project (AGENTS.md is per-repo by nature).

The installer also sets `ADLC_HOME` and adds `core/scripts` to your PATH so the `adlc` engine
resolves in any project. **Open a new terminal afterward** so PATH/ADLC_HOME take effect. (A
`adlc.cmd` shim makes `adlc` callable from cmd/PowerShell too, not just Git Bash.)

> Prefer to keep it per-project instead of global? Point the host at a project-local copy ‚Äî e.g.
> put `.clinerules/` or `.gemini/` or `.claude/` in the repo. The instruction files are identical.

## Configure (optional)

Jira is optional. With no config the framework uses **local ticket files** (`docs/adlc/tickets/`)
and the rest of the pipeline is identical. For real Jira Cloud, copy `.env.example` ‚Üí `.env` or
export `JIRA_BASE_URL`, `JIRA_EMAIL`, `JIRA_API_TOKEN`, `JIRA_PROJECT_KEY`.

## Use

| Host | How to start |
|------|--------------|
| Claude Code | `/adlc "let users reset their password by email"` |
| Cline | invoke the `adlc` workflow with your request |
| Gemini CLI | `/adlc "..."` |
| AGENTS.md tools | tell the agent to "run the ADLC pipeline for: ..." |

Then follow the two prompts (approve the spec, approve the push). Re-run with `resume <KEY>` to
continue from `state.md`.

## The deterministic engine (works with zero LLM)

```bash
adlc init "add a health endpoint"     # -> ADLC-001, seeds docs/adlc/ADLC-001/
adlc next-key                          # next local key
adlc jira mode|pick|create ...         # dual-mode ticketing
adlc get-state <KEY> <field> | set-state <KEY> <field> <value>
adlc approve <KEY> gate1|gate2         # record a human approval
adlc compare-url <branch>              # PR compare URL from origin
adlc ship <KEY> [--no-push]            # commit (+push) + print compare URL
```

## Extending

- Change behavior in `core/`, run `py build.py`, commit. Never edit `adapters/`.
- Add a host: write a `build_<host>()` in `build.py` that emits that host's config from `core/`.
- Later: wrap `core/scripts` in an MCP server for MCP-capable hosts (structure is ready).

## Out of scope (for now)
- Automated GitHub **PR creation** (shipper prints the compare URL; open the PR manually).
- Dedicated Cursor/Windsurf adapters (covered by `AGENTS.md`).
- MCP server layer and CI wiring.
